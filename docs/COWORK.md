# Working with next-caltrain-pwa

This repo started as a Caltrain schedule PWA for KaiOS phones and is also
becoming the canonical tool for validating and publishing the schedule used
by the iOS (`next-caltrain-swift`) and Android (`next-caltrain-kotlin`)
sibling apps — see ROADMAP.md "Repo mission shift". The PWA itself
(`webapp/`) isn't going away or becoming legacy: it stays live at
https://next-caltrain-pwa.appspot.com/, and browsing it against a candidate
`schedule.json` is itself the manual/visual validation step before that
schedule gets published. The published `schedule.json` is fetched at runtime
by all three clients, including the PWA.

## Repo structure

- `webapp/` — the PWA itself (HTML/JS/CSS, deployed to App Engine as static
  files, live at https://next-caltrain-pwa.appspot.com/); doubles as the
  manual validation tool — browse it against a candidate schedule before
  publishing (see ROADMAP.md)
- `src/` — JS source including `CaltrainService.js`, the reference routing implementation
- `data/` — schedule CSVs, flat (not yet organized into date folders — see
  ROADMAP.md); weekday/weekend are GTFS-generated, holiday is hand-maintained
  (see "Schedule data pipeline" below)
- `generate.py` — generates `weekday_*.csv`/`weekend_*.csv` from Caltrain's
  GTFS feed via partridge; also writes GTFS-derived holiday output to
  `data/_holiday_*.csv` shadow files (comparison only, not authoritative)
- `update_pwa.py` — parses the CSVs and produces `schedule.json`
- `app.rb`, `lib/scrape.rb`, `lib/status.rb` — Ruby/Sinatra backend serving
  `/alerts`, `/delays`, `/scrape`; deployed alongside the static `webapp/` via
  the same `app.yaml`

## Schedule data pipeline

### Current approach
`update_pwa.py` reads six CSVs directly from `data/` and produces
`src/@caltrainServiceData.js`:
- `weekday_north.csv`, `weekday_south.csv`
- `weekend_north.csv`, `weekend_south.csv`
- `holiday_north.csv`, `holiday_south.csv`

The weekday/weekend CSVs are generated by `generate.py` from Caltrain's GTFS
feed and have been verified byte-for-byte identical to the hand-maintained
versions they replaced.

`holiday_north.csv`/`holiday_south.csv` remain hand-maintained and
authoritative. Caltrain's GTFS feed's holiday/modified-schedule service has
confirmed data-quality problems (see "Holiday schedule" below), so
`generate.py` does not overwrite these two files — it writes its GTFS-derived
holiday output to `data/_holiday_north.csv`/`data/_holiday_south.csv`
instead, as shadow files for ongoing comparison only.

There's no `data/YYYY-MM-DD/` folder structure yet and no automated, date-gated
deploy — that's planned (see ROADMAP.md) but not built. Today, updating the
weekday/weekend schedule means re-running `generate.py`; updating the holiday
schedule means hand-editing the CSVs (see below); either way, finish by
running `update_pwa.py` and deploying manually (see "Deploying" below).

`next-caltrain-swift` also has its own `tools/convert_schedule.py`, which does
the same CSV → `schedule.json` conversion and is the script actually used in
the documented cross-repo publish workflow (it writes straight into this
repo's `webapp/schedule.json`; see swift's `docs/GIT_WORKFLOW.md`). This
repo's `update_pwa.py` is redundant with it today. Folding both into one
implementation that lives here is part of the mission shift in ROADMAP.md.

### Holiday schedule: hand-maintained via AI-assisted PDF transcription
Caltrain publishes a "Modified Schedule" timetable PDF covering a handful of
upcoming holiday/modified-schedule days at a time (e.g.
https://www.caltrain.com/media/36013/, covering Nov 28 2025, Dec 24 2025, and
Jan 19 2026). Each time Caltrain publishes a new one, have AI fetch and
read the PDF and update `holiday_north.csv`/`holiday_south.csv` by hand to
match it exactly — station list, train numbers, and every departure time.
This should recur a few times a year, not on every schedule change.

As of 2026-06-17, the hand-maintained files were checked against the PDF
above and matched it exactly except for one gap, since fixed: Tamien was
missing the two South County Connector departures (train 819 at 6:44a, train
825 at 8:14a) northbound.

The GTFS-derived `_holiday_*.csv` shadow files, by contrast, have several
confirmed errors against the same PDF: wrong times at San Jose Diridon
(trips 141/143 northbound, 104/108 southbound), a spurious College Park
entry (trip 143 northbound) and a missing one (trip 139 northbound), and —
most visibly — spurious Tamien/Capitol/Blossom Hill/Morgan Hill/San
Martin/Gilroy stop times attached to roughly 19 trains each that don't
actually serve those stations per the official timetable (only 819/825 do).
This is consistent with the previously-noted MLK-2027 GTFS date bug — GTFS's
holiday/modified-schedule data isn't trustworthy yet. Re-check the shadow
files against a fresh PDF next time one is processed, and only switch
`update_pwa.py`/`generate.py` over to the GTFS version once it stops
disagreeing with Caltrain's own published timetable.

### Target approach (see ROADMAP.md)
The remaining gap is the date-folder/auto-deploy pipeline and a validation
layer described in ROADMAP.md — the GTFS parsing itself is done for
weekday/weekend.

### Published endpoint
`https://next-caltrain-pwa.appspot.com/schedule.json`

The `scheduleDate` field in the JSON is an epoch-ms timestamp (newest source
file mtime) used by clients as a freshness/version marker.

The PWA itself is live at the same App Engine app's root,
https://next-caltrain-pwa.appspot.com/ — it's an actively published app, not
just a hosting shell for the JSON.

## CaltrainService.js — reference implementation

`src/CaltrainService.js` is the canonical routing logic. When debugging
schedule or routing issues in the iOS/Android apps, compare their
`CaltrainService.swift` / `CaltrainService.kt` against the JS equivalents:
`direction()`, `select()`, `times()`, `merge()`.

## Running locally

Frontend (vanilla JS/HTML/CSS — there is no `npm start`; it's static):
```bash
npm install
npm run build     # babel-transpiles src/ into webapp/script.js
open webapp/index.html
```

Backend (Ruby/Sinatra — `/alerts`, `/delays`, `/scrape`):
```bash
bundle exec ruby app.rb
open http://localhost:4567/alerts
```

## Deploying

```bash
npm run build    # rebuild webapp/script.js from src/
npm run deploy   # gcloud app deploy app.yaml --quiet — ships webapp/ + the
                  # Ruby backend to App Engine, immediately, no date gate
```
