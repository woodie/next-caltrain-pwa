# feed/

`schedule.json` here is the live schedule file, regenerated by
`update_json.py` on every publish. `app.yaml` serves it at
`/feed/schedule.json` — the canonical URL for all current and future app
versions.

There's a second, separate file, `webapp/schedule.json`, served at the
legacy `/schedule.json` URL — a hand-frozen copy held specifically for the
iOS 1.0 App Store review build. This pipeline never writes to it. See
`docs/PUBLISHING.md` ("Schedule JSON URLs" section) for the full picture.

This directory lives at the repo root, next to `webapp/`, not inside it —
and deliberately isn't named `data`. `.gcloudignore` excludes the
pipeline-only `/data` directory at the repo root; an earlier version of that
exclusion pattern was unanchored and also matched `webapp/data/`, which
silently dropped this file from deploys. Avoid reintroducing a directory
named `data` anywhere in the deploy path.

The `/feed/schedule.json` handler sets `expiration: "10m"` in `app.yaml`,
which bounds how long an App Engine edge cache can keep serving a stale
copy after a new schedule is published.

Don't hand-edit this file — it's generated by `update_json.py` and checked
by `validate_schedule.py`. See `docs/PUBLISHING.md` ("Schedule JSON URLs"
section) for the full pipeline and rationale.
